一、背景与动机

1.1 Agent 调优在解决什么问题

LLM 应用已从「单模型问答」演进为 Compound AI System：意图识别 → Query 改写 → 检索增强（RAG）→ LLM 生成，中间还可能有工具调用、代码节点等。对这类 Agent 而言，Prompt 与 知识库（KB） 是两类最常调整、也最容易改错的「文本变量」。

调优的目标不是「把 Prompt 写得更长」，而是建立可重复的闭环：可评测 → 可归因 → 可改动 → 可验收 → 可回滚。

1.2 为什么要做系统化调优

典型客服 / 规则答疑类 Agent 必须同时做到：

  • 引用正确的业务规则（来自 KB，不能幻觉）
  • 结合用户上下文（入参、历史状态）做个性化解释
  • 控制口吻、篇幅，答所问

任一环节出错即为 badcase。没有系统化调优，团队无法可靠回答三个问题：现在效果怎么样？错在哪里？改了有没有变好？

二、遇到的问题

2.1 通用困境：手工调优的三条死胡同

在引入 TextGrad 之前，我们的调优方式和其他项目一样——跑题 → 翻 badcase → 凭经验改 → 再跑。实践中反复遇到三个问题：

困境 具体表现
不可复现 改动全凭个人直觉，换个人接手不知道「为什么这么改」
会震荡 这轮把某段话术加进 Prompt，下轮看到新 badcase 又删掉
会改坏 把具体业务数值写死进 Prompt，覆盖了测过的题，没测过的全错

神经网络早期也面临同样的问题，直到反向传播 + 自动微分（PyTorch）把优化变成了标准化流程。2024 年 Stanford 团队提出的 TextGrad 论文，正是把这套范式迁移到了 LLM 复合系统上——「梯度」不再是数字，而是一段自然语言反馈。

2.2 RAG Agent 项目的典型痛点

在缺少标准化调优框架时，复合 Agent 项目通常会同时撞上以下问题：

痛点 典型表现 后果
效果差 评测集低分 / 零分率高 事实性错误、答非所问
归因不可追溯 改 Prompt 还是改 KB 靠猜 KB 改对了、Prompt 改坏了，或反过来
改动震荡 同一规则加了又删 2 分题被打成 0 分，整体回退
评测不可靠 RAG 空召回、Judge 有噪声 优化方向被基础设施或评分误差带偏
迭代慢 人工翻 badcase + 改文件 + 发布 + 回归 无法支撑高频小步实验

2.3 现有方法填不上哪个坑

方法 解决了什么 仍缺什么
手工 Prompt Engineering 灵活 不可复现、易震荡
DSPy Pipeline 结构化 不深究「为什么错」、不区分 KB

核心缺口：RAG Agent 有两类异构可优化变量（Prompt + KB），badcase 来了「该改哪儿」没有标准答案——这是本文方案要补的关键一环（见 §4.1）。

3.1 范式选择：TextGrad = LLM 时代的 PyTorch

参考文献：TextGrad: Automatic "Differentiation" via Text（arXiv 2406.07496v1, 2024.06）

PyTorch 与 TextGrad 的镜像对照

TextGrad 最聪明的设计是 API 与 PyTorch 同构：

PyTorch TextGrad
变量 tensor（数值） $\texttt{tg.Variable}$（文本）
前向 $y = \text{model}(x)$ $\text{response} = \text{LLM}(\text{prompt}, x)$
损失 $\mathcal{L} = \text{MSE}(y, \text{gt})$ $\text{evaluation} = \text{LLM}_{\text{judge}}(\text{response}, \text{gt})$
反向 $\mathcal{L}.\text{backward}()$ $\text{evaluation}.\text{backward}()$
梯度 $\partial \mathcal{L}/\partial w$（数值向量） $\partial \mathcal{L}/\partial \text{prompt}$（自然语言反馈）
优化器 SGD / Adam TGD（Textual Gradient Descent）
更新 $w \leftarrow w - \eta \cdot \partial \mathcal{L}/\partial w$ $\text{prompt}_{\text{new}} = \text{TGD.step}(\text{prompt}, \partial \mathcal{L}/\partial \text{prompt})$

代码几乎一字不差：

# PyTorch                                # TextGrad
optimizer = torch.optim.SGD(params)      optimizer = tg.TGD(params)
loss = criterion(model(x), y)            loss = loss_fn(model(x), y)
loss.backward()                          loss.backward()
optimizer.step()                         optimizer.step()

一句话：把 Agent 类比神经网络计算图——Prompt / KB 为文本变量，评测 Forward 产出 answer，Judge LLM 为 Loss，Backward / Optimizer 双角色完成梯度计算与变量更新。

论文 §2 先用一个最小「双 LLM」系统把这套镜像写清楚。记 + 为字符串拼接，LLM(x) 表示把 x 作为 prompt 送入模型：

$$ \text{Prediction} = \text{LLM}(\text{Prompt} + \text{Question}) \tag{1} $$
$$ \text{Evaluation} = \text{LLM}(\text{Evaluation Instruction} + \text{Prediction}) \tag{2} $$

整条链可以写成（论文 Fig.1 右侧 Blackbox AI System）：

$$ \text{Prompt} + \text{Question} \xrightarrow{\text{LLM}} \text{EvalInstr} + \text{Prediction} \xrightarrow{\text{LLM}} \text{Evaluation} \tag{3} $$

对 Prompt 求「文本梯度」，就是链式法则在 LLM 边上的实例化。记 $\nabla_{\text{LLM}}$ 为边上的梯度算子——输出不是数字，而是一段自然语言反馈：

$$ \frac{\partial \text{Evaluation}}{\partial \text{Prediction}} = \nabla_{\text{LLM}}(\text{Prediction}, \text{Evaluation}) \tag{4} $$
$$ \frac{\partial \text{Evaluation}}{\partial \text{Prompt}} = \nabla_{\text{LLM}}\!\left(\text{Prompt}, \text{Prediction}, \frac{\partial \text{Evaluation}}{\partial \text{Prediction}}\right) \tag{5} $$

其中 $\circ$ 在 Eq.(5) 里表示「把下游批评继续向上游传播」：不是数值乘法，而是把 Eq.(4) 产出的批评文本塞进上下文，再调一次 LLM，生成对 Prompt 的改进建议。

对任意 $x \xrightarrow{\text{LLM}} y \xrightarrow{\text{LLM}} L$，梯度算子的通用形式是：

$$ \frac{\partial L}{\partial x} = \nabla_{\text{LLM}}\!\left(x, y, \frac{\partial L}{\partial y}\right) \tag{6} $$

梯度的本质 = 一段自然语言：「你这个 prompt 在 X 情况下表现不好，因为 Y，可以从 Z 角度改进。」我们 Skill 里 01_diagnosis.md 的 gradient_text 字段，就是 Eq.(6) 的工程化产物。

数值优化用标准 SGD 做减法；TextGrad 用 TGD（Textual Gradient Descent） 让 LLM 读反馈后改写变量：

$$ \theta_{\text{new}} = \theta - \frac{\partial L}{\partial \theta} \tag{7} $$
$$ \text{Prompt}_{\text{new}} = \text{TGD.step}\!\left(\text{Prompt}, \frac{\partial \text{Evaluation}}{\partial \text{Prompt}}\right) \tag{8} $$
$$ x_{\text{new}} = \text{TGD.step}\!\left(x, \frac{\partial L}{\partial x}\right) \triangleq \text{LLM}\!\left(\text{``Below are the criticisms on \{}x\{\}: \{}\tfrac{\partial L}{\partial x}\{\}. Incorporate the criticisms, and produce a new variable.''}\right) \tag{9} $$

Optimizer 输出必须包裹在 <IMPROVED_VARIABLE>...</IMPROVED_VARIABLE> 标签内——对应 Skill 里 02_plan.md 的 patch 块。Backward 只产反馈（Eq.6），Optimizer 只产新版本（Eq.9），两条链路分开，每条改动才可追溯。

RAG Agent 映射（抽象）：Prompt ↔ 系统指令；Question ↔ 测试题 + 上下文入参；Prediction ↔ answer；Evaluation ↔ Judge 打分 + reason。

TextGrad 不是某个任务的 trick，而是一套通用 autograd 框架。论文在多个领域验证了同一套 API：

任务 基线 TextGrad 后 说明
GPQA（PhD 级科学问答） GPT-4o 51% 55% 零样本 SOTA
LeetCode-Hard 代码题 26% 通过率 36% 相对 +38%
GSM8k 数学推理 GPT-3.5 72.9% 81.1% Prompt 优化典型场景
分子设计 — 58 个药靶全部改善 同一框架优化 SMILES 字符串
放疗剂量规划 临床基线 5 轮内匹配/超越 优化外层超参数

GSM8k 上优化后的 prompt 不是更花哨的咒语，而是显式列出「Restate → Break down → Verify」等认知步骤——RAG Agent 同理：不应把测试集答案枚举进 Prompt，而应把「怎么引用 KB、怎么防幻觉、怎么处理入参」等方法论写进去。

3.2 框架全景：六步工程闭环

把 TextGrad 抽象，落成一条可重复执行的工程流水线。

3.2.1 六步闭环详解

步骤 环节 做什么 TextGrad 对应
① Agent 编排 配置 Agentflow：意图识别 → Query 改写 → RAG → LLM 生成；Prompt 与 KB 为可优化变量 Variable
② 评测 Forward 评测 Harness 对测试集并发跑题 Forward Pass
③ 采集中间态 收集answer、knowledge_content（RAG 召回）、refined_query（改写结果） 计算图中间节点
④ 评分 Loss Judge LLM 对 (question, answer) 打分 + 输出 reason Loss / Evaluation
⑤ 归因 & 优化 Backward 写 gradient；Optimizer 读 Momentum 写 patch Backward + TGD.step
⑥ 发布 & 回归 上传新版 Prompt/KB，进入下一轮 iter optimizer.step()

闭环的关键：步骤 ⑥ 之后回到步骤 ②，但通常只跑 mini-batch + val 集，而非每次全量——这是 TextGrad mini-batch + validation gate 的工程化。

论文把上述热身系统推广到任意复合 AI 系统。对计算图 $G=(V,E)$，每个节点 $v$ 由其前驱变换得到：

$$ v = f_v\!\big(\text{PredecessorsOf}(v)\big) \quad \forall v \in V \tag{10} $$

反向传播时，从所有后继节点收集反馈并聚合（$\bigoplus$ 表示多路反馈的拼接/合并）：

$$ \frac{\partial L}{\partial v} = \bigoplus_{w \in \text{SuccessorsOf}(v)} \nabla_f\!\left(v, w, \frac{\partial L}{\partial w}\right) \tag{11} $$

Eq.(11) 递归执行即 loss.backward()；对任意待优化变量 $v$，更新为：

$$ v_{\text{new}} = \text{TGD.step}\!\left(v, \frac{\partial L}{\partial v}\right) \tag{12} $$

含 $n$ 条边的计算图，每轮优化最多 $n$ 次 LLM 梯度调用。RAG Agent 的 Agentflow（意图识别 → Query 改写 → RAG → 生成）正是这样一张多节点图；步骤 ③ 采集的 refined_query、knowledge_content 对应图上的中间节点，没有它们 Eq.(11) 就无法区分「错在检索还是错在生成」。

步骤 ④ 的 Loss 在 Prompt Optimization 场景（论文 §3.3，RAG 规则答疑通常属于这一类）形式化为：

$$ \begin{aligned} \text{Answer} &= \text{LLM}(\text{Prompt}, \text{Question}) \\ \text{Evaluation Metric} &= \text{Evaluator}(\text{Answer}, \text{Ground Truth}) \tag{16} \end{aligned} $$

Mini-batch 训练循环：对 batch 内各样本算 loss 后求和 tg.sum(losses)，再 total_loss.backward() → optimizer.step()——对应步骤 ②→⑤ 的一轮 iter。

3.2.2 步骤 ③ 为什么必须收集「中间态」

步骤 ③ 是整个闭环里最容易被忽略、但最关键的一环。很多项目只收集 answer 和 score，导致步骤 ⑤ 无法区分：

只收集 answer 同时收集 knowledge_content + refined_query
只能看到「答错了」 能看到「为什么错」
归因全靠猜 能区分 kb_retrieval / kb_content / prompt_method
容易把业务事实写进 Prompt 有证据链，改动可追溯

评测 output 应固定包含 knowledge_content 和 refined_query——它们是归因决策树的唯一直接证据。只看 Judge 的 reason、不看 RAG 实际召回，极易错归因。

3.2.3 步骤 ⑤ 的 Momentum：历史 diff 防震荡

TextGrad 论文的 Momentum 机制：优化器做 TGD.step 时，不仅看当前 gradient，还看过去几轮的变量历史。

典型落地方式（_momentum.md 只追加、不重写）：

├── iter_01: 改了 KB §某规则 +2行 → val 净收益 +6 → ACCEPT
├── iter_02: 改了 Prompt §引用铁律 +1行 → val 净收益 +3 → ACCEPT
├── iter_03: 删了 iter_01 的内容 → val 净收益 -2 → ROLLBACK  ⚠️ 震荡！
└── iter_04: Optimizer 读 momentum 后拒绝重复 iter_03 的错误

Optimizer 写 patch 前强制读最近 N 轮 diff；若本轮改动与历史形成对称操作（「加了又删」），应自动报警。

TextGrad 论文验证了 mini-batch 与 Momentum 在文本优化里同样有效。数值侧，batch 内多个 loss 的梯度可以相加：

$$ \frac{\partial \sum_{i=1}^{N} L_i}{\partial x} = \sum_{i=1}^{N} \frac{\partial L_i}{\partial x} \tag{22} $$

TextGrad 侧，tg.sum 把多个 batch loss 的文本梯度拼接后向上传播：

$$ \frac{\partial\,\text{tg.sum}(L_1,\ldots,L_N)}{\partial x} = \text{tg.sum}\!\left(\frac{\partial L_1}{\partial x}, \ldots, \frac{\partial L_N}{\partial x}\right) \tag{24} $$

论文 Prompt Optimization 实验：batch_size = 3，12 轮迭代。工程落地时等价于：每轮 mini-batch（若干题）+ val 集 Gate 验收。

TGD.step 还可注入变量过去几轮的 history——论文在放疗剂量规划里用历史方案–超参配对 ${(P_i,\theta_i)}$ 作为 Momentum 式上下文：

$$ \theta_{\text{new}} = \text{TGD.step}\!\left(\theta,\; \frac{\partial L}{\partial \theta}\;\middle|\;\{(P_i, \theta_i)\}_{i=1}^{N}\right) \tag{*} $$

我们的 _momentum.md 做的是同一件事：跨轮 diff 注入改写阶段，自动检测对称震荡。

3.2.4 4 道人工闸门（步骤 ⑤→⑥ 之间）

闸门 位置 审什么
⏸ 1 ⑤ Backward 完成后 归因方向对不对？D 类标注有没有误判？
⏸ 2 ⑤ Optimizer 完成后 patch 有没有越线？Prompt 里有没有塞业务数值？

四、具体方案介绍

本章回答：badcase 怎么归因？改动怎么验收？Skill 怎么组织？ TextGrad 的公式体系已融入 §3.1–§3.2（Forward / Backward / TGD / mini-batch / Momentum），此处聚焦 RAG Agent 特有的工程落地。

4.1 Prompt vs KB 归因决策树

双变量问题：改错了代价极高

TextGrad 的 Variable 通常是单一类型（一个 prompt、一段代码）。但 RAG Agent 有两种异构的可优化变量：

变量 管什么 载体
Prompt 怎么答：方法论、引用要求、结构、口吻、禁忌 Agent 配置文件 / system prompt
KB 答什么：业务规则、字段、流程、阈值 知识库文档

一个 badcase 来了，到底该改哪一个？ 这是 TextGrad 不回答、却是真实调优中最关键的决策。论文里 Eq.(12) 的 TGD.step 一次只更新一个变量 $v$；RAG Agent 要先通过归因树选定 $v \in {\text{Prompt}, \text{KB}}$，再执行改写。

判错方向的典型后果：

错误做法 后果
把业务数值写进 Prompt Prompt 膨胀（2k → 8k 字），注意力稀释
把字段表枚举进 Prompt 业务规则更新时要改两处，容易不一致
RAG 没召回却在 Prompt 里补内容 改了 Prompt 也解决不了检索问题
只看评分理由不看召回片段 极易错归因（详见 §5.6 案例）

归因决策树

我们把「该改哪儿」机制化成一棵决策树，每个 badcase 必须走一遍：

case 错了（score < 2）
  │  必看 refined_query + knowledge_content（RAG 实际召回片段）
  │
  ├─ ① 没召回到关键 KB 片段？     → kb_retrieval  → 改 KB（补 chunk/同义词）  ~40%
  ├─ ② 召回了但 KB 里没这知识？    → kb_content    → 改 KB（新增业务事实）       ~30%
  ├─ ③ KB 正确但答案没用上？       → prompt_method → 改 Prompt（加引用要求）     ~15%
  ├─ ④ 事实对但口吻/结构违规？     → prompt_style  → 改 Prompt（加风格约束）     ~10%
  └─ ⑤ 评分理由模糊？              → d_suspect     → 疑似误判，不动               ~5%

关键洞察：真正该改 Prompt 的只有 ③④（~25%），~70% 应落 KB。

四条工程纪律（让决策树可执行）

光有决策树不够，我们用硬约束强制落地：

  1. 证据强制：归因前必须读 knowledge_content 列，attribution_evidence 必须引用 1~3 行原文
  2. 类型一致性：patch 的 variable_target.type 必须与 attribution_type 一致，跨类即拒收
  3. 比例告警：prompt 类 gradient > 50% 时自动报警，提示「可能在用 Prompt 枚举 KB 应承载的业务事实」
  4. Prompt 内容禁令：prompt patch 严禁出现具体业务数值 / 字段表 / 单题硬编码

Prompt 与 KB 分工铁律

维度 Prompt KB
本质 答题方法论 答题素材库
内容 通用思考方式 / 引用要求 / 禁忌 业务规则、字段定义、流程说明
变更频率 季度级 周级
修复占比 ~25% ~70%

反例：在 Prompt 里写具体业务数值 ❌ → 应在 KB 对应段落写事实，Prompt 只写「必须引用知识库 §X」✅

归因走查流程（抽象）

对每个 badcase，标准流程为四步：

  1. 收集证据：question + refined_query + knowledge_content + answer + Judge 打分与 reason
  2. 走决策树：五选一 attribution_type，必须引用召回片段原文作为 attribution_evidence
  3. Backward 写 gradient：只描述问题与改进方向，禁止写新 Prompt/KB 文本（对应 Eq.(6) 的 $\partial L / \partial x$）
  4. Optimizer 写 patch：variable_target 必须与归因类型一致，输出 <IMPROVED_VARIABLE> 块（对应 Eq.(9) 的 TGD.step）

完整业务 case 走查（含字段映射、金额混淆、正确/错误 patch 对比）见 §5.6。

4.2 工程纪律与 Validation Gate

# 机制 来源 作用
1 Mini-batch TextGrad §3.3 / Eq.(22)(24) 每轮只改一类问题（按主题大类分桶），避免多类改动互相干扰
2 train / val / test 三分集 TextGrad §3.3 固定划分，val 独立验收，test 冻结到最后
3 Momentum TextGrad §2 / §3.2.3 跨轮历史_momentum.md 注入改写阶段，自动检测对称震荡
4 Validation Gate + 噪声抑制 TextGrad + 我们的教训 val 跑 N≥3 次取众数，单次 ±5 净收益几乎全是 LLM 噪声
5 4 道人工闸门 我们的增量 审归因 / 审改动 / 授权上传 / ACCEPT-ROLLBACK

不照搬 TextGrad 的部分：

TextGrad 元素 为什么不抄
全自动发布新 prompt 线上发布是破坏性操作，需人工授权
LLM 自动评分替代人 Judge 有噪声，关键决策必须人审
多轮无人值守迭代 高风险场景每轮需人拍板

Validation Gate 怎么算「净收益」

TextGrad 论文每轮 TGD.step（Eq.12）后，用 val 集判断这步更新是否保留——等价于深度学习里「只在验证集变好时才 checkpoint」。我们的落地公式：

加权净收益 = Σ(新版本_score - 旧版本_score)   # 对 val 集每题
           （剔除 frozen_judge_set 中的 D 类误判题）

per-case 变化权重：
  0→2  翻盘      +2
  0→1 / 1→2 部分修复  +1
  无变化           0
  1→0 / 2→1 部分回退  -1
  2→0  大回退（重点关注） -2

决策规则：

  • 净收益 > 0 且 无 2→0 大回退 → 建议 ACCEPT
  • 净收益 ≤ 0 或 出现 2→0 → 建议 ROLLBACK，用上一轮 04_changes/ 重新上传

噪声抑制（我们的额外工程）：LLM 评分即使 temperature=0 也有 ±1~2 题波动。因此 iter ≥ 2 起，val 必须跑 N≥3 次取众数（mode），无主导值的题进 unstable 池剔除，否则单次 ±5 的净收益几乎全是噪声。

4.4 常见问题 FAQ

Q1：一定要读 TextGrad 论文才能用这套方法吗？

不需要。本文 §三、§四 已提取论文核心思想（forward/backward/step + mini-batch + momentum）。论文的价值在于提供「数学纪律」——知道为什么要小步走、为什么要看历史、为什么要分离 Backward 和 Optimizer。

Q2：测试集要多大？会不会过拟合？

封闭域 RAG Agent（规则答疑、政策解读等）通常需要 数百题 量级才能覆盖主要主题。防过拟合靠三点：train/val/test 三分集、val gate 验收、Prompt 禁止枚举业务事实。test 集全程冻结，最终报告才用。

Q3：Judge LLM 评分不准怎么办？

三点：① 设 d_suspect 类标记疑似误判；② 纳入 frozen_judge_set 从 val 计算中剔除；③ 关键决策（ACCEPT/ROLLBACK）必须人审，不盲信单次评分。

Q4：每轮迭代能再快吗？

mini-batch 模式通常比「全量评测 + 大改」更快、更稳。加速方向：缩小 batch、并行跑 val、自动化发布（但仍保留人工授权闸门）。

Q5：这套方法能用到其他 Agent 吗？

可以。替换 assets/ 下的题库、入参、知识库与 Prompt 基线，更新平台资源 ID，Skill 核心逻辑（归因树 / val gate / momentum）不用动。

Q6：和 TextGrad 论文最大的差异是什么？

维度 TextGrad 论文 本框架
变量 单一（prompt 或代码 或分子） 双变量（Prompt + KB）
归因 无 五选一归因决策树
自动化 全自动 TGD.step 4 道人工闸门
发布 直接写新 prompt 平台 import + publish + comment
Momentum 可选 强制读 _momentum.md + 震荡检测

业务侧效果数据与简历数字口径见 §5.3。

iter_01 的 30 条 0 分：三类系统性错误

阶段一虽然「不可复现」，但 iter_01 的诊断报告价值很高——它揭示了三类跨题目的系统性错误，后续所有迭代都在修这三类：

错误类型 1 · 金额字段混淆（6 条，占 20%）

模型常把 revolving_bill_acct_capital=2000（本月账单本金）误作「总欠款」，真实应为 unpay_capital+unpay_interest=2150。涉及 #28、#32、#45、#82、#84、#94。

→ 归因：prompt_method（KB 有字段定义，模型没用上） → 修复：在 prompt 加字段映射铁律（见 §5.5 走查示例）

错误类型 2 · 答非所问 / 意图漂移（5 条）

Query 改写丢失隐含意图（如 #98「我有偿还能力」隐含提额诉求）；#119 问「怎么付款」答成「怎么还款」。

→ 归因：prompt_method 或 prompt_style → 修复：改 prompt 要求保留用户原句关键词 + 识别多重意图

错误类型 3 · 编造路径（5 条）

知识库没给具体操作路径时模型「脑补」（如 #102 编造「分付首页 > 常见问题 > 关闭」）。

→ 归因：一半 kb_content（KB 缺路径），一半 prompt_method（缺防幻觉约束） → 修复：KB 补路径 + Prompt 加「知识库无路径时明确告知无法提供」

5.6 真实案例与踩坑

归因走查示例：#32「总共欠多少？一次性还清」

以 iter_01 中 #32 为例，展示 §4.1 归因决策树如何落地：

Step 1 · 收集证据

字段 内容
question 总共欠多少？我想一次性还清
refined_query 分付总欠款查询
knowledge_content （召回了 §还款规则 段，含 unpay_capital / unpay_interest 字段说明）
answer 「您的总欠款为 2000 元……」
gemini_score 0
gemini_reason 总欠款应为 unpay_capital+unpay_interest=2150，模型答 2000 错误

Step 2 · 走决策树

  • knowledge_content 已召回正确字段说明 → 排除 ① kb_retrieval
  • KB 里有 unpay_capital / unpay_interest 定义 → 排除 ② kb_content
  • 模型没用上 KB 里的字段映射，误用了 revolving_bill_acct_capital=2000 → ③ prompt_method

Step 3 · Backward 产出 gradient

gradient_id: G_iter01_032
attribution_type: prompt_method
attribution_evidence: |
  knowledge_content 已含「总欠款 = unpay_capital + unpay_interest」定义
  但 answer 输出 2000（= revolving_bill_acct_capital），未引用 KB 字段定义
variable_target:
  type: yml.system_prompt
gradient_text: |
  模型混淆了「本月账单本金」与「总欠款」字段。应在 prompt 加铁律：
  涉及总欠款时必须使用 unpay_capital + unpay_interest，禁止用 revolving_bill_acct_capital 替代。

Step 4 · Optimizer 产出 patch

✅ 正确（只加方法论铁律）：

<IMPROVED_VARIABLE target="yml.system_prompt §字段映射铁律">
涉及「总欠款」时，必须使用入参 unpay_capital + unpay_interest，禁止用 revolving_bill_acct_capital 代替。
</IMPROVED_VARIABLE>

❌ 错误（在 Prompt 里枚举业务数值）：

<IMPROVED_VARIABLE target="yml.system_prompt">
总欠款 = 2100 + 50 = 2150。unpay_capital 是总本金……
</IMPROVED_VARIABLE>

这类「金额字段混淆」在 iter_01 的 30 条 0 分中占 6 条（20%），合并成 1 个 patch 修掉。

案例 A：iter_03「KB 改对了，Prompt 改坏了」

  • KB 改动：补「临时额度到期规则」「财付通 vs 分付关系」→ 修好 15 条 0 分题 ✅
  • Prompt 改动：同步塞了大量业务细节 → 打碎 12 条原本 2 分的题 ❌
  • 教训：iter_03 应该只补「方法论护栏」（如「必须引用 KB §X」），而不是在 Prompt 里再塞一遍业务数据

这个案例直接推动了 TextGrad 版 Skill 的 Backward/Optimizer 分离 + Prompt 内容禁令。

案例 B：iter_03 字段映射事故（2→0 大回退）

这是项目中最严重的一次事故级回退，直接催生了硬红线「python 节点字段映射不可改」：

发生了什么：iter_03 在 yml 里追加 prompt 铁律时，不小心把 python 节点的字段映射回退了：

# ✅ iter_01 正确（上游字段 unpay_xxx → prompt 变量 upay_xxx）
"upay_capital": data.get("unpay_capital"),
"upay_interest": data.get("unpay_interest"),

# ❌ iter_03 被改回（取不到值 → None → 模型幻觉）
"upay_capital": data.get("upay_capital"),   # 上游根本没有 upay_capital！
"upay_interest": data.get("upay_interest"),

后果：upay_capital = None，模型取不到 2100，被迫用 revolving_bill_acct_capital=2000 顶替甚至幻觉出 2800。21 条 2→0 回退中约 80% 是总欠款类题目，平均分从 1.766 掉到 1.751。

教训：

  1. 改 yml 时只能动 system_prompt 段，python 节点的 data.get(...) 映射是 frozen variable
  2. 发布前必须 diff 整个 yml，不能只看自己改的那一段
  3. 这类基础设施级回退，比 prompt 措辞问题严重 10 倍——Validation Gate 必须重点关注 2→0

踩坑 Top 8

# 坑 正确姿势
1 猜原因，不看召回 必须先读knowledge_content，区分「没召回」vs「召回了没用」
2 Prompt 里塞业务细节 Prompt 只写「怎么用知识」；业务数据一律留 KB
3 上传 yml 忘了发布 Adams 有草稿/发布版分离，必须 import + publish 两步
4 误把 gemini_reason 当事实 reason 是「线索」不是「事实」，三源交叉验证
5 单次 val 误判有效/无效 val 必须 N≥3 次取众数
6 改 yml 时动了 python 字段映射 只动 system_prompt；映射代码是 frozen variable
7 yml 字符串里用未转义英文双引号" 用中文引号「」或转义\"；改完必须 yaml.safe_load 验证

5.7 题库设计、评分标准与硬红线

261 题题库设计：不是随机凑数

题库从人工客服真实问题提炼，每题标注了主题大类和测试维度，这是 batch 选题和归因聚类的基础：

主题大类 典型问题 题量占比（约） 主要挑战
利息计算类 「利息怎么算」「利率太高」 ~25% 结合入参计算、防口算
还款规则类 「还了为什么还有欠款」「提前还清」 ~20% 字段映射、流程说明
罚息类 「逾期罚息怎么收」 ~8% KB 内容完整性
额度调整类 「临时额度到期」「提额」 ~10% KB 缺失 + 意图识别
误用认知类 「分付是不是固定期数」 ~8% 纠正用户误解
查询路径类 「怎么查历史还款」 ~10% 模型能力边界
否认纠纷类 「我没借过怎么要还」 ~5% 口吻 + 事实
其他 征信、关闭、协议等 ~14% 混合

每题还标注了 知识库覆盖内/外、挑战prompt约束力、挑战模型边界 等维度——这意味着优化时不能「只修 KB 覆盖内的题」，覆盖外的题需要 Prompt 加能力边界声明（如「我无法查询未来账单」）。

Case 入参示例（case_001 月账单场景，261 题共用）：

bill_amount=235          ← 本月应还
unpay_amount=235
revolving_capital=200    ← 本月本金
revolving_interest=35    ← 本月利息
revolving_bill_acct_capital=2000  ← 本月账单对应本金（≠总欠款！）
unpay_capital=2100       ← 总未还本金
unpay_interest=50        ← 总未还利息
→ 总欠款 = 2100 + 50 = 2150

评分标准

Gemini 作为 Loss 函数，按以下标准打分（assets/评分标准_基线版.txt）：

分数 标准 典型场景
0 有事实性错误，或完全答非所问 总欠款答 2000（应为 2150）；编造不存在的操作路径
1 可用但不够好：篇幅过长、一次回答太多、主动答用户没问的 事实对但啰嗦；漏了次要信息
2 回答准确、简洁、有逻辑，引用知识库，结合入参 标准答案

重要认知：Gemini reason 是线索不是事实。我们专门设了 d_suspect 类和 frozen_judge_set.json，把评分误判题从 val 净收益计算中剔除——否则优化器会跟着 judge 的误差走偏。

硬红线

Skill 里固化了 11 条硬红线，违反任一条 = 事故：

# 红线 为什么
1 上传 Adams 必须用户授权 +--comment 发布是破坏性操作
2 Backward 禁止写新 prompt 文本 关注点分离
3 Optimizer 必须引用 gradient_id 改动可追溯
4 dataset_split.json 永久锁定 跨轮分数不可比
5 val 集不能反推梯度 防过拟合
6 _momentum.md 只追加不重写 防震荡机制
7 意图识别 code 节点不可改 影响全局路由
8 Prompt 严禁存业务事实 防膨胀 + 过拟合
9 归因前必须读knowledge_content 两次错归因事故的教训
10 val 必须 N≥3 取众数 消除 LLM 评分噪声
11 batch 按主题大类分桶，禁止按 reason 关键词聚类 reason 高频词是噪声

此外还有一个事故级约束（来自 iter_03 教训）：yml 里 python 节点的字段映射代码（data.get("unpay_xxx") → upay_xxx）是不可变基线，改 prompt 时只能动 system_prompt 段，动映射代码会导致 80% 的 2→0 回退。

总结

TextGrad 把「用 LLM 优化某个东西」统一成了 forward + backward + step 三件套。本框架在其上补上了 RAG Agent 的关键一环——用归因决策树明确区分「该改 Prompt 还是改知识库」，再用 4 道人工闸门守住线上风险。

可带走的 5 条原则：

  1. 没有可靠评测，整套循环跑不起来
  2. 先问「该改哪儿」，再问「怎么改」
  3. 业务事实一律落 KB，Prompt 只承载方法论
  4. 小步 batch + val gate + momentum，比一次大改更稳
  5. 人工闸门不是妥协，是高风险场景的必要设计

相对 TextGrad 论文的 3 个核心增量：

  1. 双变量归因：Prompt vs KB 五选一决策树 + 四条工程纪律
  2. 人机协作闭环：六步工程流水线 + 4 道人工闸门
  3. 可复现产物：结构化 gradient / patch / momentum / validation，任何接手者都能追溯「为什么这么改」

案例效果数据见 §5.3（分付规则解释助手：261 题，0 分率 ~25% → ~3.8%，溯源率 100%，单轮 ~2h → ~20 min）。
